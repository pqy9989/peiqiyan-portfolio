## PROTO · 内容区域布局原型图
> 发散阶段产出，**直接内嵌本段**（不另建 `proto/` 子目录）。仅画 `main-content` 内容区（navbar / sidebar / tabbar 由 fragment 容器提供，不画）
> ⚠️ **`content-header` 右上角禁止画主 CTA**——主 CTA `[+ 新建 / 发起 / 提交]` 已由 navbar.slot-action 接住，PROTO 重复画就违反 fragment "唯一主 CTA" 强约束。卡片内的次级操作按钮（"一键领取" / "复用" / "撤回"）可以画
> ⚠️ **不 hardcode 业务文案**——PROTO 是发散阶段，只标区域名（`content-header` / `filter-bar` / `工作台 3 卡块`）+ 字段语义（`F1·套餐识别`）+ 关键交互锚点（`[一键领取]`）。具体文案（标题副标题 / 按钮措辞 / tab 名 / 卡片标语）由 page-agent 装配时按 REQ 的 DO/SHOW 段填，PROTO 里写 `<副标题占位>` 即可
- **方案数量**：⬜ 单选（1 个最终方案） / ⬜ 多选（N 个并行方案，N=2~4）
  - 单选：画 1 张原型在 `### option-1` 子段，下游派 **1 个 page-agent**，产出 `pages/<slug>/<slug>.{html,css}`
  - 多选：每个方案一个 `### option-<N>` 子段（ASCII + 设计意图 + F1 落位），下游**并行派 N 个 page-agent**，**同一目录**产出 `pages/<slug>/<slug>-option-<N>.{html,css,-RULES.md}`（不再按方案分子文件夹）
- **画什么**：内容区分块（header / filter / 主区 / 抽屉 or 弹窗），标注 F1 字段落位
- **怎么画**：ASCII 框图 / Mermaid `flowchart` 内嵌于代码块；**不画具体组件样式**，只表达区域比例和信息密度
- **每个方案子段固定 3 部分**：① ASCII 框图；② 设计意图（1-3 句）；③ F1 字段落位表

- **多选方案差异化要求**：N 个方案必须在**结构层面**有差异（如：列表 vs 网格 vs 工作台；抽屉填表 vs 跳新页向导），不能只换配色/字号
- **验收**：每张原型图需与本 REQ 的 F1/F2 字段、菜单结构、交互形态自洽；不一致需回头修订

## EDGE · 边界 + 约束 + 成功
- **5 种状态**：
  - 空态：（首次进入引导文案 + 主 CTA 描述，**不指定具体组件**，只描述视觉/行为意图）
  - 加载态：（骨架屏 / spinner / 分批加载——只描述形态意图）
  - 错误态：（网络异常 / 字段校验失败 / 权限不足）
  - 成功态：（toast / 跳转 / 停留）
  - 无权限态：（隐藏 / 灰化 / 提示申请）
- **私有 CSS 前缀**：`--<slug-prefix>-*`
- **视口**：（仅桌面 1440+ / 自适应 / 仅移动）
- **成功指标**：（如"85% 用户 3 分钟内完成提交"）

> ⚠️ 本节**禁止列具体 cs- 组件清单**（如"复用 button / tag / drawer"）。组件选型是 page-agent 装配时的工作，REQ 只描述业务意图和约束。

## 派单准备
- 直接派 **page-agent**，把 REQ 整段内容（含内嵌 PROTO 章节）塞进 prompt
- PROTO 多选时主会话**并行**派 N 个 page-agent（参考 `superpowers:dispatching-parallel-agents` 技能），每个 prompt 引用同一份 REQ 文件 + 指明 option 序号（如"装配 option-2，参考 PROTO 段下的 `### option-2` 子段"），同目录产出 `pages/<slug>/<slug>-option-<N>.{html,css,-RULES.md}`
- page-agent 装配时 read `design/components/INDEX.md` 评估组件复用，缺失组件在最终报告里列出，由主会话决定是否派 component-agent 新建后重装
- 多方案并行时 page-agent **禁止 read 兄弟 option 文件**，否则违反 PROTO 结构差异硬约束

C1 · fragment 容器

不修改 fragment 容器的几何 / 结构

C1.1~1.3

3 个检查点

fragment.html 是统一框架的源头，页面层修改 navbar / sidebar / content-header 的几何（padding / margin / gap / height）会让多页面节奏失控，且无法被其他页享受。

1.1
未自创 layout class（.nav-link / .nb-item / .ss-item 等）
grep "class=\"" *.html 与 fragment.css 比对
1.2
未在 page 层覆盖 fragment 容器的几何（含给槽位加 padding/margin）
grep "cs-navbar\|cs-sidebar\|cs-content-header\|.slot-" page.css
1.3
未在 .slot-left 上手写 flex-direction: column（依赖 fragment.css :has(.page-subtitle) 自动 column 逻辑）
grep "slot-left" + "flex-direction" page.css
默认间距不合用 → 反馈 fragment-agent 改 fragment.css，不在 page 层 patch
C2 · 三大区域职责切分

slot-action / slot-right / filter-bar 互斥

C2.1~2.5

5 个检查点

三大区域是 fragment 的强约束，决定页面的视觉一致性。一旦混放，主 CTA / 视图切换 / 筛选 会重复或错位，所有页面失去节奏感。

2.1
cs-navbar.slot-action 只放 1 个主 CTA（cs-button data-variant="primary"）
grep "slot-action" + count cs-button
2.2
cs-content-header.slot-right 只放搜索（cs-search-trigger / cs-search-input），禁止视图切换 / 筛选 / 新建
grep "slot-right" 内容
2.3
cs-filter-bar 内不放新建 / 主 CTA 按钮
grep "cs-filter-bar" ~ "/cs-filter-bar" 内 cs-button data-variant="primary"
2.4
视图切换 / 筛选 / 排序 都放在 cs-filter-bar，不放 content-header 右上
grep "view-switch\|filter" + 出现位置
2.5
搜索互斥：默认在 content-header；若 cs-filter-bar[data-variant="search"]，则 slot-right 留空
grep "data-variant=\"search\"" + slot-right 内容
C3 · 令牌使用

无硬编码 · 无 undefined token

C3.1~3.4

4 个检查点

硬编码会让深浅色 / Figma 变量切换失效；引用不存在的令牌（如 --radius-md 未声明）会静默 fallback 到 0，圆角 / 间距彻底失效但不报错。

3.1
无硬编码颜色（#xxx / rgb() / rgba()）
grep -E "#[0-9a-f]{3,6}|rgba?\(" page.css
3.2
无硬编码像素值（除框架透传需要的容器宽度）
grep -E "[0-9]+px" page.css 后人工核
3.3
未引用未声明的令牌（典型陷阱：--radius-md、--space-lg）
grep "var(--" page.css → 与 variable.css 比对
3.4
所有 var() 引用都能被 base.css 解析（无 fallback 到 0）
preview eval: getComputedStyle 验证关键属性非空
缺令牌 → 反馈 css-agent 在 variable.css 加；不在 page 层硬编码兜底
C4 · 图标声明

只用已声明的 cses-* class

C4.1~4.2

2 个检查点

图标字体的 class 必须在 cses-icons.css 中实际声明，否则页面渲染成空白方框（无报错）。凭印象写图标名（如 cses-stratis-check-01）是常见陷阱。

4.1
所有 cses-* class 都在 cses-icons.css 中实际声明
grep "cses-" page.html → 与 cses-icons.css 内 class 列表比对
4.2
找不到的语义图标降级为通用图标 + 颜色 / 文字区分（不留空）
人工 review 偏差表里的图标降级是否合理
缺图标 → 反馈 font-agent 在 cses-icons.css 加；本次装配先用降级图标 + 在偏差表里说明
C5 · 私有命名空间

前缀隔离 · 不污染设计系统

C5.1~5.3

3 个检查点

页面层私有变量 / class 必须用模块前缀（--meeting-room-* / .mr-*），否则会污染全局命名空间，可能与其他模块或组件冲突。

5.1
所有页内私有 CSS 变量用模块前缀（--meeting-room-* / --meeting-room-<page-slug>-*）
grep "^\s*--" page.css 后核前缀
5.2
所有自创 class 用模块前缀（.mr-* / .meet-find__* 等）
grep "^\\." page.css 后核前缀
5.3
未覆盖 fragment.css / 组件 CSS 的官方 class（cs-*）的几何
grep "cs-[a-z-]*\s*{" page.css 后人工核
C6 · 引用顺序

link 顺序固定 · 优先级正确

C6.1~6.3

3 个检查点

CSS 引用顺序决定级联优先级。颠倒会让令牌、图标、fragment 容器失效或部分失效。

6.1
link 顺序：base.css → cses-icons.css → fragment.css → page-shell.css → 组件 CSS → shell.css → 页私有.css
grep "<link" page.html 后逐行核
6.2
所有静态资源路径返回 HTTP 200
preview 加载页面，看 console / network 无 404
6.3
路径前缀正确（../../design/ 两级，相对于 pages/<module>/）
grep "href=\"\\." page.html
C7 · 跨页约束

不读兄弟 · 不复用页面 · href 接通

C7.1~7.3

3 个检查点

「不读兄弟页面」是为了避免业务结构互相污染（如所有页的卡片都变成同一样子）。但允许 read shell.html，因为它是统一框架。

7.1
未 read 兄弟页面 HTML（pages/<module>/<其它-page-slug>.html）
复盘装配过程的 Read 调用历史
7.2
多页面 Stage 2 各页 read 并克隆 shell.html（这是允许且必须的例外）
grep doctype / cs-layout / tabbar / sidebar / navbar 结构完整性
7.3
跨页 <a href> 都指向真实兄弟页（sidebar 菜单 / 列表行 / 详情按钮 / 通知项）
grep "href=\"./" page.html → 核每个目标文件存在
C8 · overlay 属性正确性

默认 closed = 不写属性

C8.1~8.4

4 个检查点 HIGH-RISK

fragment.css 用 [data-open] / [data-visible] 这种属性存在选择器。写 data-open="false" 会被 CSS 当成"打开"，抽屉默认就显示。这是高发陷阱。

8.1
cs-drawer / cs-modal 默认关闭时不写 data-open 属性（不是 data-open="false"）
grep "data-open=\"false\"" page.html → 必须 0 命中
8.2
cs-drawer-overlay 默认关闭时不写 data-visible 属性
grep "data-visible=\"false\"" page.html → 必须 0 命中
8.3
JS toggle 用 setAttribute('data-open', '') + removeAttribute('data-open')，不是 set 'true'/'false'
grep "setAttribute.*data-open" + "removeAttribute"
8.4
overlay 自身点击 / Esc / 关闭按钮 都能把属性 remove 掉
preview 实际点 / 按 Esc / 看 overlay attribute 变化
C9 · demo 数据

真实场景词 · 不字面 <占位>

C9.1~9.2

2 个检查点

字面 <占位> 让 review 时完全没法判断信息密度 / 字段长度 / 视觉重量。装配时必须用真实感的中文 demo 数据（"望京 3F-A 大会议室" / "Q3 OKR 复盘" 这种）。

9.1
所有列表行 / 详情字段 / 抽屉表单值 用真实感的中文 demo
grep "<占位>\|<F1·" page.html → 应集中在系统级标题 / 不可知运营文案
9.2
demo 数据覆盖典型场景（不同状态、不同长度、含极端值）
人工 review：是否含至少 1 个 confirmed / 1 个 cancelled / 1 个 long 主题

失败处理

FIX-RULE · 3 类失败 3 种处理

失败类型处理谁来修
A. 装配错误 page-agent 修正后重跑自检 page-agent 自己（不向主会话报错）
B. 缺资源（组件 / 图标 / 令牌） 在"偏差"里列出，使用降级方案完成本次装配 主会话决定是否派 component-agent / font-agent / css-agent 后回头重装
C. 强约束冲突（PROTO 跟 fragment 互相打架） 暂停装配，向主会话报告需要 PROTO 调整 或 fragment 调整 主会话回到需求阶段或 fragment-agent
永远不要静默忽略自检失败：所有偏差必须写进报告，让主会话知情决策。
